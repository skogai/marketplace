# contributing lessons

this guide explains how to create lessons.

## what makes a good lesson?

a good lesson:

- addresses a recurring pattern, not a one-time fix
- has clear detection signals
- provides a minimal correct example
- is concise enough to inject directly
- uses keywords/tools that match real hook context

avoid creating lessons for:

- one-time fixes that will not recur
- subjective preferences without clear benefits
- trivial patterns everyone knows
- platform-specific hacks without a reusable rule

## lesson categories

keep the current tree small. add categories only when they contain real lessons.

### concepts (`lessons/concepts/`)

core concepts and mental models.

### workflow (`lessons/workflow/`)

best practices for development workflows and processes.

future categories can include:

### tools (`lessons/tools/`)

constraints, limitations, and correct usage patterns for specific tools.

### patterns (`lessons/patterns/`)

general patterns that span multiple tools or contexts.
